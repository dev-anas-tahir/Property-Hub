## Key Implementation Patterns

### 1. Frontend Build Strategy

**Critical:** Source files (`frontend/`) are **separated** from production assets (`static/`). This prevents `collectstatic` errors with Tailwind's `@import` directives.

- **Source files**: `frontend/src/input.css` (contains `@import "tailwindcss/base"` etc.)
- **Compiled output**: `static/dist/output.css` (processed by Tailwind)
- **Django collects**: Only files from `static/`, never from `frontend/`

**Build Commands:**
```bash
# Development (watch mode)
npm run build-css

# Production (minified)
npm run build-css-prod

# Then collectstatic
python manage.py collectstatic --noinput
```

**See:** [Frontend Architecture](./docs/architecture/frontend.md) | [Frontend Setup](./docs/development/frontend-setup.md)

### 2. Django Settings Organization

Settings are split by environment:
- `config/settings/base.py` - Common settings
- `config/settings/development.py` - Development overrides
- `config/settings/production.py` - Production overrides

Environment variables loaded from `.env`, `.env.dev`, or `.env.prod` using `environs`.

**Key Settings:**
```python
# Custom user model
AUTH_USER_MODEL = "users.User"

# WebSocket channel layer
CHANNEL_LAYERS = {
    "default": {
        "BACKEND": "channels_redis.core.RedisChannelLayer",
        "CONFIG": {"hosts": [(REDIS_HOST, REDIS_PORT)]},
    }
}

# Static files with WhiteNoise
STATIC_URL = "static/"
STATIC_ROOT = BASE_DIR / "staticfiles"
STATICFILES_DIRS = [BASE_DIR / "static"]
STORAGES = {
    "staticfiles": {
        "BACKEND": "whitenoise.storage.CompressedManifestStaticFilesStorage",
    },
}
```

### 3. Component-Based Templates

Templates use a component architecture for DRY code:

```django
{% extends "_layouts/base.html" %}

{% block content %}
  {% include "_components/properties/property-card.html" with property=property %}
{% endblock %}
```

**Component locations:**
- Forms: `templates/_components/forms/`
- Navigation: `templates/_components/navigation/`
- Properties: `templates/_components/properties/`
- UI elements: `templates/_components/ui/`

### 4. WebSocket Chat Implementation

The chat system uses Django Channels with Redis as the channel layer.

**Client:** `frontend/src/chat-client.js` - `ChatClient` class
- Exponential backoff reconnection (1s, 2s, 4s, 8s, 16s)
- Message validation (1-5000 characters)
- Callback system for messages and connection status

**Server:** `apps/chat/consumers.py` - ASGI consumer
- Handles WebSocket connections
- Broadcasts messages to conversation groups
- Persists messages to database

**See:** [Chat Client Documentation](./docs/development/chat-client.md)

### 5. Database & Migrations

- **Database**: PostgreSQL 17 (development uses Docker)
- **Migrations**: Located in each app's `migrations/` directory
- **Custom user model**: `apps/users/models.py` - `User` extends `AbstractUser`

**Important:** Run migrations after creating models:
```bash
python manage.py makemigrations
python manage.py migrate
```

### 6. Static Files & Media

- **Static files**: CSS, JS, images in `static/` (compiled) and `frontend/src/` (source)
- **Media uploads**: User-uploaded files in `media/`
- **Production storage**: Optional AWS S3 for media/static via `django-storages`

**S3 Configuration:**
```python
USE_S3_MEDIA = env.bool("USE_S3_MEDIA", False)
USE_S3_STATIC = env.bool("USE_S3_STATIC", False)
```

## Security Considerations

1. **Authentication**: Custom `User` model with Django's auth system
2. **Brute-force protection**: Django Axes (10 attempts, 30min lockout)
3. **CSRF protection**: Enabled by default
4. **Password validation**: Django's validators (similarity, length, common, numeric)
5. **Session storage**: Redis (not database) for performance
6. **Secrets**: All secrets via environment variables (never hardcoded)
7. **Production**: Always set `DEBUG=False`, configure `ALLOWED_HOSTS`, enable HTTPS

**See:** [Architecture Overview - Security](./docs/architecture/overview.md#security)


## Important Notes for Agents

### Do NOT:
1. **Modify** `frontend/` files during Django `collectstatic` - they are source files, not production assets
2. **Add** source files to `STATICFILES_DIRS` - only `static/` should be collected
3. **Hardcode** secrets or environment-specific values - always use environment variables
4. **Commit** compiled `static/dist/output.css` if CI/CD builds it - but it's okay to commit for releases
5. **Edit** generated files in `staticfiles/` - this directory is auto-generated
6. **Change** `AUTH_USER_MODEL` after migrations are created - requires database migration strategy
7. **Disable** security features (CSRF, Axes) without explicit requirement and mitigation plan

### Do:
1. **Follow** the component-based template structure in `templates/_components/`
2. **Use** Tailwind utility classes first, custom CSS only when necessary
3. **Add** custom CSS in `frontend/src/input.css` within `@layer` directives
4. **Write** reusable components for repeated UI patterns
5. **Document** complex logic with comments
6. **Run** `ruff check` and `ruff format` before committing
7. **Test** frontend build and collectstatic before deployment
8. **Reference** existing documentation in `docs/` before making architectural changes
9. **Use** the `ChatClient` class for WebSocket communication (don't create raw WebSocket connections)
10. **Keep** settings in environment-specific files, not in code

## Quick Decision Guide

### Adding a new page?
1. Create view in appropriate `apps/<app>/views.py`
2. Add URL pattern in `apps/<app>/urls.py`
3. Create template in `templates/<app>/` using `_layouts/base.html`
4. Reuse components from `templates/_components/` as needed

### Adding new styles?
1. Use Tailwind utility classes directly in HTML
2. For reusable styles, add to `frontend/src/input.css` in `@layer components`
3. Run `npm run build-css` (dev) or `npm run build-css-prod` (prod)

### Adding JavaScript interactivity?
1. For simple UI: Use Alpine.js (already included via DaisyUI)
2. For dynamic content: Use HTMX (add to template)
3. For complex logic: Add to `frontend/src/` and build to `static/dist/`
4. For WebSocket: Use `ChatClient` class (see `frontend/src/chat-client.js`)

### Need real-time features?
1. Use Django Channels (see `apps/chat/` as reference)
2. Define consumer in `consumers.py`
3. Add WebSocket routing in `config/routing.py`
4. Use `ChatClient` pattern on frontend

### Storing files?
1. **User uploads**: Use `MEDIA_ROOT` (or S3 if `USE_S3_MEDIA=True`)
2. **Static assets**: Place in `static/` (images, fonts) or `frontend/src/` (CSS/JS to compile)
3. **Never** store files in `frontend/` that should be collected by Django

### Changing database schema?
1. Update model in `apps/<app>/models.py`
2. Run `python manage.py makemigrations`
3. Run `python manage.py migrate`
4. Update forms, views, and templates as needed
5. Add admin configuration in `apps/<app>/admin.py`

### Need admin interface?
1. Register models in `apps/<app>/admin.py`
2. Use Django Unfold for modern UI (already configured)
3. Customize with `list_display`, `search_fields`, `list_filter`, etc.
